# Requirements: Repo Context Packer v2.1 Pack Inspector & GraphRAG Retrieval

**Defined:** 2026-05-18
**Milestone:** v2.1 Pack Inspector & GraphRAG Retrieval
**Core Value:** Given a coding task, ctxpack should return the smallest safe evidence set that makes an existing coding agent more likely to inspect the right files, run the right tests, and avoid irrelevant context.

## v2.1 Requirements

Requirements for a diagnostic pack inspector, retrieval-health views,
GraphRAG-style code retrieval, embedding controls, and agent context previews.

### Inspector Foundation

- [x] **INSP-01**: Maintainer can generate a source-free `PackInspectorView` for a context pack that exposes pack identity, task metadata, budget, target agent, token estimate, privacy status, warnings, and diagnostics.
- [x] **INSP-02**: Maintainer can inspect target files, related tests, validation commands, selected memory, and pack sections with evidence attribution and without losing original `ContextPlan` and `ContextPack` provenance.
- [x] **INSP-03**: Maintainer can export inspector-ready JSON and Markdown/HTML preview artifacts from the CLI without starting a persistent UI process.
- [x] **INSP-04**: Inspector outputs clearly separate source-bearing safe snippets from source-free report metadata and include privacy flags that make this boundary visible.

### Local Web Inspector

- [x] **UI-01**: Maintainer can start or export a local read-only web inspector that loads pack inspector artifacts and renders the current pack without editing source files or mutating global agent configuration.
- [x] **UI-02**: User can view target files, related tests, validation commands, evidence reasons, signal scores, warnings, selected memory, and budget allocation in a dense diagnostic layout.
- [x] **UI-03**: User can filter and search inspector content by file path, repo label, evidence signal, warning type, candidate role, and selected memory source.
- [x] **UI-04**: Inspector UI has responsive desktop and narrow-width layouts verified with automated UI smoke tests, and does not require daily usage for normal CLI/MCP workflows.

### Retrieval Health

- [x] **HEALTH-01**: Maintainer can generate a retrieval-health report that summarizes recent benchmark trends, historical eval recall, repeated retrieval gap families, and token ROI from source-free local records.
- [x] **HEALTH-02**: Retrieval-health views explain signal contribution across lexical, symbol, graph, semantic, history, tests, memory, feedback, active paths, and workspace routing.
- [x] **HEALTH-03**: Retrieval-health output highlights low-confidence subsystems, repeated missing file/test families, and policy profiles that improved or degraded measured outcomes.
- [x] **HEALTH-04**: Health reports remain source-free by default and do not include raw prompts, source snippets, terminal logs, or model transcripts.

### GraphRAG Retrieval

- [x] **GRAPH-01**: Maintainer can generate source-free graph neighborhood views around a task, file, symbol, package/module, workspace repo, memory card, or retrieval candidate.
- [x] **GRAPH-02**: ctxpack can compute source-free module/community summaries over existing repository graph edges, dependency edges, tests, memory cards, feedback events, and workspace boundaries.
- [x] **GRAPH-03**: Graph-aware retrieval policy experiments can add graph neighborhood/community candidates as measurable signals without changing default pack ranking until eval evidence supports it.
- [x] **GRAPH-04**: Graph expansion remains budgeted and non-recursive by default, with diagnostics when expansion is capped, degraded, or unavailable.

### Embedding Controls

- [x] **EMBED-01**: Maintainer can inspect semantic provider status, vector metadata counts, provider kind, model ID, dimensions, indexing freshness, and whether semantic retrieval was used in a plan or pack.
- [x] **EMBED-02**: Maintainer can compare local semantic, lexical-only, graph-only, hybrid, and disabled-semantic retrieval policies in benchmark and feedback reports.
- [x] **EMBED-03**: Cloud embedding or reranking provider configuration is explicit, policy-gated, disabled by default, and reflected in privacy status and inspector output.
- [x] **EMBED-04**: Embedding provider/dimension changes require source-free eval metadata so policy changes can be audited and rolled back.

### Agent Previews

- [x] **AGPREV-01**: User can preview how a pack or context plan would be presented to Codex, Claude Code, Cursor, OpenCode, and a generic MCP client.
- [x] **AGPREV-02**: Agent previews show which MCP tools/resources, AGENTS.md guidance, native rules, and pack resource URIs an agent should use for a task.
- [x] **AGPREV-03**: Agent previews preserve the product boundary: ctxpack suggests context and validation strategy, while the target agent remains responsible for reading files, editing, and running commands.
- [x] **AGPREV-04**: Preview artifacts are source-free unless the user explicitly exports a source-bearing pack preview from safe inventory snippets.

## Out of Scope

Explicitly excluded from v2.1 to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Daily coding UI | ctxpack remains agent-native; the UI is diagnostics and preview, not an editor replacement. |
| Autonomous code editing | ctxpack remains a read-only context broker; agents own edits and permissions. |
| Hidden cloud embeddings or reranking | Any cloud provider must be explicit, policy-gated, and disabled by default. |
| Tauri desktop packaging as the first UI deliverable | A local web/static inspector proves utility before desktop packaging complexity. |
| Generic LLM-built document GraphRAG | v2.1 uses ctxpack's existing code graph, semantic metadata, memory, feedback, and workspace signals. |
| Full large-repo graph rendering by default | Inspector graph views should render subgraphs, communities, and focused evidence neighborhoods. |
| Hosted backend, SSO, or enterprise admin | Useful later, but v2.1 focuses on local diagnostics and measured retrieval improvements. |
