# Roadmap: Repo Context Packer

## Overview

This roadmap opens v1.7 Adaptive Retrieval Policy & Feedback Loop. v1.2 gave ctxpack benchmark proof, v1.3 made storage durable, v1.4 added optional semantic retrieval, v1.5 added parser/precision overlays, and v1.6 added source-free repo memory. v1.7 uses those foundations to learn from source-free agent-session feedback, report policy quality, propose safe retrieval-policy changes, and prove whether context budgets improve outcomes.

## Phase Details

### Phase 30: Feedback Event Contracts & Ingestion

**Goal**: Source-free feedback events capture what agents actually read, edit, test, pass, fail, or receive corrections for.

**Depends on**: v1.2 eval traces, v1.3 storage, v1.6 memory cards

**Requirements**: LEARN-01, LEARN-02, LEARN-03, LEARN-04

**Success Criteria** (what must be TRUE):
1. Feedback event contracts serialize with stable source-free public JSON shapes.
2. Feedback ingestion records recommended/read/edited/tested/outcome/correction metadata without raw prompts, source snippets, terminal logs, secrets, or cloud payloads.
3. Feedback event storage is local, schema-versioned, deterministic, and compatible with older stores.
4. CLI output can record, list, and summarize recent feedback events in Markdown and JSON.

**Plans**: 1 plan

Plans:
- [x] 30-feedback-event-contracts-ingestion-01-PLAN.md — Add feedback contracts, local ingestion, storage diagnostics, and focused tests.

### Phase 31: Policy Quality Reports

**Goal**: Maintainers can generate source-free policy reports for context precision, token ROI, repeated gaps, and validation coverage.

**Depends on**: Phase 30

**Requirements**: LEARN-05, LEARN-06, LEARN-07, LEARN-08

**Success Criteria** (what must be TRUE):
1. Reports compare recommended files/tests against read, edited, tested, passed, failed, blocked, and corrected feedback.
2. Reports include context precision, read precision, edit recall proxy, validation coverage, repeated missing-file families, signal contribution, and token ROI where metadata exists.
3. Reports can group by task type, target agent, pack budget, retrieval signal, and repository area without storing source text or prompts.
4. Low-sample or degraded evidence is labeled explicitly.

**Plans**: 1 plan

Plans:
- [x] 31-policy-quality-reports-01-PLAN.md — Add policy report computation, renderers, diagnostics, and source-free tests.

### Phase 32: Adaptive Policy Profiles & Rollback

**Goal**: ctxpack can propose, inspect, apply, disable, and roll back local retrieval-policy profiles from evidence.

**Depends on**: Phase 31

**Requirements**: LEARN-09, LEARN-10, LEARN-11, LEARN-12

**Success Criteria** (what must be TRUE):
1. Candidate policy profiles are generated from benchmark and feedback evidence without mutating the active policy by default.
2. Candidate profiles include rationale, before/after metrics, regression warnings, sample counts, and rollback metadata.
3. Exact path, symbol, test, privacy, and active-anchor signal floors are enforced.
4. CLI output can inspect, apply, disable, and roll back local policy profiles in JSON and Markdown.

**Plans**: 1 plan

Plans:
- [x] 32-adaptive-policy-profiles-rollback-01-PLAN.md — Add policy profile proposal, persistence, safety floors, and rollback controls.

### Phase 33: Agent Outcome Comparison

**Goal**: Maintainers can compare plan-only, brief, standard, and deep pack outcomes across fixed tasks without source leakage.

**Depends on**: Phase 32

**Requirements**: LEARN-13, LEARN-14, LEARN-15, LEARN-16

**Success Criteria** (what must be TRUE):
1. Outcome comparisons aggregate source-free feedback and eval metadata by pack budget and task set.
2. Reports include pass rate, blocked rate, correction rate, validation-command coverage, average context size, and useful target files per 1k estimated tokens.
3. Comparisons identify changed-sample and low-information-task risks before claiming lift.
4. Product proof includes v1.7 feedback and outcome sections without source leakage.

**Plans**: 1 plan

Plans:
- [x] 33-agent-outcome-comparison-01-PLAN.md — Add budget comparison reports and product-proof integration.

### Phase 34: Feedback Docs, External Smoke, and Release Gates

**Goal**: Documentation and release gates prove feedback learning is local, source-free, measurable, and backward-compatible.

**Depends on**: Phase 33

**Requirements**: LEARN-17, LEARN-18, LEARN-19, LEARN-20

**Success Criteria** (what must be TRUE):
1. Docs explain feedback events, reports, tuning profiles, rollback, privacy guarantees, and anti-patterns.
2. Release smoke proves source-free ingestion, report generation, candidate tuning, rollback metadata, and outcome comparison.
3. RefactoringMiner or another large-history repo smoke uses fixed sample boundaries before claiming improvement.
4. Existing prepare-task, get-pack, MCP, memory, semantic, precision, and storage behavior remains backward-compatible by default.
