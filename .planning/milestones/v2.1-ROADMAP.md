# Roadmap: Repo Context Packer

## Overview

This roadmap opens v2.1 Pack Inspector & GraphRAG Retrieval. v2.0 completed the
workspace and team layer. v2.1 turns ctxpack's accumulated retrieval evidence
into inspectable diagnostic surfaces and adds measured graph/embedding retrieval
improvements without changing the product boundary: ctxpack remains local-first,
read-only, and agent-native.

## Phase Details

### Phase 39: Inspector Contracts & Static Export

**Goal**: Maintainers can generate source-free inspector view models and static export artifacts for packs.

**Depends on**: v1.2 benchmark/proof reports, v1.6 memory, v1.7 feedback, v2.0 workspace packs

**Requirements**: INSP-01, INSP-02, INSP-03, INSP-04

**Success Criteria** (what must be TRUE):
1. `PackInspectorView` exposes pack identity, task metadata, target agent, budget, token estimate, privacy status, warnings, diagnostics, sections, selected memory, related tests, validation commands, and evidence attribution.
2. Inspector view models preserve links back to `ContextPlan`, `ContextPack`, repo/workspace IDs, candidate IDs, and source-free provenance fields.
3. CLI export can write inspector JSON and Markdown/HTML preview artifacts without starting a persistent UI process.
4. Source-bearing safe snippets and source-free report metadata are explicitly separated and privacy-labeled.

**Plans**: 1 plan

Plans:
- [x] 39-inspector-contracts-static-export-01-PLAN.md — Add inspector contracts, compiler export, CLI commands, docs, and source-free tests.

### Phase 40: Local Web Pack Inspector

**Goal**: Users can inspect pack decisions in a local read-only responsive web UI.

**Depends on**: Phase 39

**Requirements**: UI-01, UI-02, UI-03, UI-04

**Success Criteria** (what must be TRUE):
1. A local inspector command serves or exports a read-only web inspector from inspector artifacts.
2. The UI renders target files, related tests, validation commands, evidence reasons, signal scores, warnings, selected memory, and budget allocation in a dense diagnostic layout.
3. Users can filter by file path, repo label, evidence signal, warning type, candidate role, and selected memory source.
4. Automated UI smoke tests verify desktop and narrow-width layouts, no overlapping core text, no source sentinel leakage, and no dependency on the UI for normal CLI/MCP workflows.

**Plans**: 1 plan

Plans:
- [x] 40-local-web-pack-inspector-01-PLAN.md — Add the local/static inspector UI, filtering, responsive layout, and UI smoke tests.

### Phase 41: Retrieval Health Reports

**Goal**: Maintainers can inspect benchmark trends, retrieval gaps, token ROI, and signal contribution.

**Depends on**: Phase 39

**Requirements**: HEALTH-01, HEALTH-02, HEALTH-03, HEALTH-04

**Success Criteria** (what must be TRUE):
1. Retrieval-health reports summarize benchmark trends, historical eval recall, repeated gap families, and token ROI from source-free local records.
2. Reports explain contribution from lexical, symbol, graph, semantic, history, tests, memory, feedback, active paths, and workspace routing signals.
3. Reports identify low-confidence subsystems, repeated missing file/test families, and policy profiles that improved or degraded measured outcomes.
4. Reports remain source-free and include sentinel tests proving no raw prompts, source snippets, terminal logs, or model transcripts are emitted.

**Plans**: 1 plan

Plans:
- [x] 41-retrieval-health-reports-01-PLAN.md — Add retrieval-health contracts, report generation, inspector integration, docs, and smoke coverage.

### Phase 42: Graph Neighborhoods & Communities

**Goal**: Maintainers can generate source-free graph neighborhoods and module/community summaries.

**Depends on**: v1.5 precision edges, v1.6 memory, v1.7 feedback, v2.0 workspace metadata

**Requirements**: GRAPH-01, GRAPH-02, GRAPH-04

**Success Criteria** (what must be TRUE):
1. Graph neighborhood views can start from tasks, files, symbols, package/module labels, workspace repos, memory cards, or retrieval candidates.
2. Module/community summaries are derived from existing repository graph edges, dependency edges, tests, memory cards, feedback events, and workspace boundaries without source leakage.
3. Graph expansion is budgeted and non-recursive by default.
4. Diagnostics identify capped, degraded, stale, missing, or unavailable graph inputs.

**Plans**: 1 plan

Plans:
- [x] 42-graph-neighborhoods-communities-01-PLAN.md — Add graph view contracts, source-free community summaries, CLI/MCP resources, and graph diagnostics.

### Phase 43: Graph-Aware Policy & Embedding Controls

**Goal**: Maintainers can compare graph/semantic retrieval policies and inspect embedding provider controls.

**Depends on**: Phase 41, Phase 42

**Requirements**: GRAPH-03, EMBED-01, EMBED-02, EMBED-03, EMBED-04

**Success Criteria** (what must be TRUE):
1. Graph-aware retrieval policy experiments add graph neighborhood/community candidates as measurable signals without changing default ranking.
2. Semantic provider status reports expose provider kind, model ID, dimensions, vector metadata counts, indexing freshness, and per-plan/per-pack semantic usage.
3. Benchmark and feedback reports compare local semantic, lexical-only, graph-only, hybrid, and disabled-semantic policies.
4. Cloud embedding/reranking configuration remains explicit, policy-gated, disabled by default, privacy-labeled, and auditable with rollback metadata.

**Plans**: 1 plan

Plans:
- [x] 43-graph-policy-embedding-controls-01-PLAN.md — Add graph-aware policy experiments, semantic provider status, eval comparison, policy gates, and release smokes.

### Phase 44: Agent Preview & Inspector Release Gates

**Goal**: Users can preview agent-specific context usage, and release gates prove the inspector remains local, source-safe, and agent-native.

**Depends on**: Phases 39-43

**Requirements**: AGPREV-01, AGPREV-02, AGPREV-03, AGPREV-04

**Success Criteria** (what must be TRUE):
1. Users can preview how a plan or pack appears to Codex, Claude Code, Cursor, OpenCode, and generic MCP clients.
2. Previews show MCP tools/resources, AGENTS.md guidance, native rules, pack resource URIs, and recommended agent next steps.
3. Previews preserve the boundary that ctxpack suggests context and validation while agents own reading, editing, and command execution.
4. Inspector, health, graph, embedding, and agent-preview release smokes prove source-free metadata paths, explicit source-bearing pack paths, no hidden cloud use, and single-repo/workspace compatibility.

**Plans**: 1 plan

Plans:
- [x] 44-agent-preview-inspector-release-gates-01-PLAN.md — Add agent previews, docs, compatibility guards, final inspector smokes, and release-gate wiring.
