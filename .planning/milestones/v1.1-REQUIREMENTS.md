# Requirements: ctxhelm v1.1 Packaging & Adoption

**Defined:** 2026-05-13
**Milestone:** v1.1 Packaging & Adoption
**Core Value:** Given a coding task, ctxhelm should return the smallest safe evidence set that makes an existing coding agent more likely to inspect the right files, run the right tests, and avoid irrelevant context.

## v1.1 Requirements

Requirements for turning the completed v1 local context broker into an installable, documented, and smoke-testable product.

### Release Identity And Packaging

- [x] **PKG-01**: User can install a versioned `ctxhelm` binary from a release artifact and verify it with `ctxhelm --version` and `ctxhelm --help` without a source checkout.
- [x] **PKG-02**: Maintainer can build repeatable release artifacts and SHA-256 checksums from a clean checkout using a documented script or workflow.
- [x] **PKG-03**: Workspace package metadata, release tag, binary version output, README, and license metadata are consistent for the released version.
- [x] **PKG-04**: User has documented source-build fallback paths using locked dependencies, including `cargo install --git ... --tag ... --locked` and local checkout installation.
- [x] **PKG-05**: Release artifact contents are audited to exclude ctxhelm local state, traces, request logs, temp homes, secrets, absolute local paths, and unintended caches.

### Agent Setup And Adoption

- [x] **ADPT-01**: User can run `ctxhelm init` and receive a clear report of repo-local files written, skipped, or left unchanged, plus exact next setup and smoke steps.
- [x] **ADPT-02**: Generated Codex, Claude Code, Cursor, and OpenCode setup guidance stays thin, repo-local or copy/paste-oriented, and does not mutate global agent configuration by default.
- [x] **ADPT-03**: Agent setup guidance includes explicit `repo` usage, progressive `prepare_task` -> native file reads -> `get_pack` flow, and pack-resource session-scope caveats.
- [x] **ADPT-04**: User can validate generated setup artifacts and snippets for syntax, expected files, command availability, absolute binary path guidance, and no large static context injection.
- [x] **ADPT-05**: User can complete a first-pack quickstart from install to `init` to deterministic MCP proof to `prepare-task`/`get-pack` on a real repo.

### Smoke And Release Gates

- [x] **SMOKE-01**: Maintainer can run a release gate that smokes the installed or extracted `ctxhelm` binary, not only `cargo run`.
- [x] **SMOKE-02**: Deterministic MCP protocol smoke can run from a wrong cwd with an explicit repo and a selectable `CTXHELM_BIN` or equivalent binary path.
- [x] **SMOKE-03**: Optional Codex CLI and Claude Code real-client smokes record machine-checkable `prepare_task` and `get_pack` evidence with exact client versions when required.
- [x] **SMOKE-04**: Release gate checks CLI help/version, repo-local init, generated adapter artifacts, deterministic MCP protocol behavior, artifact audit, and docs command consistency.

### Documentation And Support

- [x] **DOCS-01**: README provides a short install-to-first-pack path that matches current CLI flags and does not rely on a source checkout for normal users.
- [x] **DOCS-02**: Documentation covers install methods, PATH failures, absolute MCP binary paths, `CTXHELM_HOME`, uninstall/state cleanup, wrong-cwd behavior, and common MCP startup failures.
- [x] **DOCS-03**: Documentation includes an agent setup matrix for Codex CLI, Claude Code, Cursor, and OpenCode with write scope, smoke support, and verified client versions.
- [x] **DOCS-04**: Documentation distinguishes deterministic protocol proof from real-client proof and does not claim unsupported Cursor/OpenCode real-client tool-call validation.

## Future Requirements

Deferred to future milestones. These remain intentionally out of v1.1.

### v1.2 Retrieval Quality Proof

- Broaden real-repo historical evals and benchmark reports.
- Report token ROI, retrieval deltas, failure modes, and regression trends.
- Use RefactoringMiner and additional real repositories as bounded proof targets.

### v1.3 Production Storage

- Add durable local storage for inventory/search/history when measured performance and cache discipline justify it.
- Improve incremental indexing and cache invalidation without changing privacy defaults.

### v1.4 Local Semantic Retrieval

- Add optional local embeddings/vector retrieval with hybrid fusion and explicit privacy controls.
- Keep cloud embeddings/reranking opt-in and visibly separate from local defaults.

### v1.5 Parser/Semantic Precision

- Expand parser-backed coverage or optional SCIP/LSP precision only where eval gaps show lexical/symbol/graph retrieval is insufficient.

### v2.0 Workspace And Team Layer

- Support multi-repo workspaces and source-free shared context cards.
- Add team policy files without requiring hosted source indexing.

### v2.1 UI / Pack Inspector

- Add optional diagnostics UI for packs, retrieval gaps, evals, and context health while keeping agents as the daily surface.

## Completed v1 Requirements

The previous v1 milestone is complete and remains the foundation for v1.1. It validated:

- CLI, MCP, and public JSON compatibility guardrails.
- Fresh safe inventory, centralized privacy/source-read policy, diagnostics, and constrained local write behavior.
- Typed, attributed retrieval candidates, fixed-budget ranking, historical eval metrics, signal ablations, and source-free gap reports.
- Deterministic MCP protocol checks, session-scoped pack-resource diagnostics, bounded MCP pack cache behavior, and real Codex CLI / Claude Code smoke wrappers.

## Out of Scope

Explicitly excluded from v1.1 to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Autonomous source edits by ctxhelm | Existing coding agents own editing, permissions, approvals, and shell execution. |
| Running user project test commands automatically | ctxhelm should recommend verification commands; the agent or user should run them under their permission model. |
| Global agent config mutation by default | Hidden MCP registration affects unrelated projects and weakens trust. |
| Cloud indexing, cloud embeddings, cloud reranking, telemetry, or update checks | Local-first privacy is part of the product contract. |
| Retrieval-quality changes | v1.1 is packaging/adoption; measured retrieval expansion belongs in v1.2. |
| Durable storage migration | Production storage belongs in v1.3. |
| Semantic/vector retrieval | Local semantic retrieval belongs in v1.4. |
| Broad parser/runtime migration | Parser precision belongs in v1.5 and should follow measured gaps. |
| Standalone chat/editor product or GUI | The product center remains agent-native context delivery. |
| Hosted team backend, SSO, or enterprise admin | Team/workspace capabilities belong in v2. |
| Crates.io, Homebrew, signed installers, or self-update as required v1.1 work | Useful later, but first release should prove one reliable binary artifact path. |
